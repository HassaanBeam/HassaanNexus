## Purpose

This guide provides mental models to elicit comprehensive project requirements, design, and task breakdowns through collaborative questioning. Use these to guide users through thoughtful project planning.

**KEY PRINCIPLE**: Let the user PICK the mental model(s) they want to use. Don't just select for them!

---

## How to Use This Guide (IMPORTANT!)

### Step 1: Offer Mental Models to User

After understanding the project basics, ALWAYS offer 2-3 relevant mental models and let the user pick:

**Example**:
```
Now let's dive into requirements. I have a few approaches we could use to
think through this comprehensively:

1. **First Principles**: Break down the problem to its fundamentals
   - Best for: Novel projects, challenging assumptions, getting to the core

2. **Stakeholder Mapping**: Identify everyone affected and their needs
   - Best for: Projects with multiple parties, organizational work

3. **SWOT Analysis**: Analyze strengths, weaknesses, opportunities, threats
   - Best for: Strategic projects, competitive situations

Which approach sounds most useful for your project? Or we could combine them!
```

### Step 2: User Picks

User responds with their choice (e.g., "Let's do First Principles and Stakeholder Mapping")

### Step 3: Apply the Questions

Use the questions from the selected mental model(s) to guide the elicitation conversation.

### Step 4: Adapt as Needed

If the chosen model isn't yielding good insights, offer to switch: "This model isn't quite clicking. Want to try [Alternative Model] instead?"

---

## Golden Rule

üéØ **ALWAYS offer options. NEVER just select and use a mental model without user input!**

The user should feel in control of how we approach the planning. We're guides, not dictators!

## Complete Mental Models Catalog

This catalog includes **30+ mental models** organized into **8 categories**. Use this as a reference when offering models to users.

### 1. Cognitive Models (4)
**Purpose**: Fundamental thinking frameworks for breaking down complex problems

**Models Available**:
- **First Principles** - Strip assumptions, identify fundamental truths
- **Systems Thinking** - Analyze interdependencies and feedback loops
- **Analogous Reasoning** - Apply patterns from similar domains
- **Lateral Thinking** - Generate creative alternatives through non-linear thinking

### 2. Collaborative Models (3)
**Purpose**: Multi-perspective analysis and structured team problem-solving

**Models Available**:
- **Six Thinking Hats** - Explore six distinct perspectives (facts, emotions, caution, optimism, creativity, process)
- **MECE Principle** - Mutually Exclusive, Collectively Exhaustive analysis
- **Stakeholder Mapping** - Identify all affected parties and their interests

### 3. Diagnostic Models (4)
**Purpose**: Root cause identification and problem decomposition

**Models Available**:
- **Root Cause Analysis (5 Whys)** - Drill down to fundamental causes
- **Fishbone Diagram** - Visual cause-effect analysis across dimensions
- **Pre-Mortem Analysis** - Identify potential failure modes before implementation
- **Force Field Analysis** - Identify driving and restraining forces

### 4. Strategic Models (5)
**Purpose**: Long-term planning and competitive positioning

**Models Available**:
- **Scenario Planning** - Multiple future state modeling
- **OODA Loop** - Observe-Orient-Decide-Act rapid decision cycle
- **Jobs to Be Done** - Outcome-focused needs analysis
- **Blue Ocean Strategy** - Create uncontested market space
- **PESTLE Analysis** - Political, Economic, Social, Tech, Legal, Environmental factors

### 5. Analytical Models (5)
**Purpose**: Data-driven decision-making and quantitative analysis

**Models Available**:
- **Decision Matrix** - Multi-criteria evaluation with weighted scoring
- **SWOT Analysis** - Strengths, Weaknesses, Opportunities, Threats
- **Cost-Benefit Analysis** - Quantified impact analysis with ROI
- **Pareto Analysis (80/20)** - Prioritization based on impact
- **Assumption Testing** - Identify and validate critical assumptions

### 6. Creative Models (4)
**Purpose**: Innovation, ideation, and breakthrough thinking

**Models Available**:
- **Design Thinking** - Empathize-Define-Ideate-Prototype-Test
- **SCAMPER** - Substitute-Combine-Adapt-Modify-Put to use-Eliminate-Reverse
- **Morphological Analysis** - Systematic exploration of solution dimensions
- **Blue Ocean Strategy** - Value innovation and market creation

### 7. Operational Models (4)
**Purpose**: Process optimization and efficiency improvement

**Models Available**:
- **Kanban Thinking** - Visualize work, limit WIP, optimize flow
- **Value Stream Mapping** - End-to-end process flow with waste identification
- **OKR Framework** - Objectives and Key Results alignment
- **Lean Canvas** - One-page business model for rapid planning

### 8. Validation Models (4)
**Purpose**: Testing assumptions and validating solutions

**Models Available**:
- **Hypothesis Testing** - Scientific method with controlled experiments
- **Prototyping** - Build testable representations for early validation
- **Pre-Mortem Analysis** - Imagine failure to prevent it
- **Red Team Analysis** - Adversarial perspective to find weaknesses

## Mental Model Selection Matrix

### For Requirements.md

**Use when:**
- Understanding project scope and goals
- Identifying stakeholders and constraints
- Uncovering assumptions and risks

**Recommended Mental Models:**

#### 1. **First Principles Thinking**
**Purpose**: Break down complex problems to fundamental truths

**Questions to Ask:**
- What's the absolute core problem we're solving?
- If we stripped away all assumptions, what remains?
- What are the fundamental requirements (not nice-to-haves)?
- What would this look like if we designed from scratch?

**When to Use**: Complex projects, unclear scope, or when user is copying existing solutions

---

#### 2. **SWOT Analysis**
**Purpose**: Understand internal and external factors

**Questions to Ask:**
- Strengths: What advantages do you have for this project?
- Weaknesses: What challenges or gaps exist?
- Opportunities: What external factors could help?
- Threats: What could prevent success?

**When to Use**: Strategic projects, competitive situations, resource planning

---

#### 3. **Stakeholder Mapping**
**Purpose**: Identify all affected parties and their interests

**Questions to Ask:**
- Who will benefit from this project?
- Who will use the deliverables?
- Who needs to approve or provide input?
- Who might be affected negatively?
- What does each stakeholder need from this project?

**When to Use**: Projects with multiple parties, organizational change, client work

---

#### 4. **PESTLE Analysis**
**Purpose**: Understand external macro-environmental factors

**Categories:**
- **Political**: Regulations, compliance, governance
- **Economic**: Budget, funding, market conditions
- **Social**: User behavior, cultural factors, trends
- **Technological**: Tools, platforms, technical constraints
- **Legal**: Contracts, intellectual property, liability
- **Environmental**: Sustainability, physical workspace

**Questions to Ask:**
- Which of these factors apply to your project?
- What external constraints exist?
- What opportunities does the environment provide?

**When to Use**: Business strategy, market entry, compliance-heavy projects

---

#### 5. **Assumption Testing**
**Purpose**: Uncover and validate hidden assumptions

**Questions to Ask:**
- What are you assuming is true about this project?
- What if that assumption is wrong?
- How can we validate this assumption early?
- What's the riskiest assumption?

**When to Use**: Innovative projects, unproven concepts, high-risk initiatives

---

### For Design.md

**Use when:**
- Planning implementation approach
- Making architectural decisions
- Breaking down complex systems

**Recommended Mental Models:**

#### 1. **Systems Thinking**
**Purpose**: Understand how parts interact within the whole

**Questions to Ask:**
- What are the major components of this solution?
- How do these components interact?
- What feedback loops exist?
- What are the inputs and outputs of each component?
- What happens if one component fails?

**When to Use**: Complex projects, interconnected systems, technical architecture

---

#### 2. **Decision Matrix**
**Purpose**: Evaluate multiple options objectively

**Questions to Ask:**
- What are the possible approaches?
- What criteria matter most? (cost, time, quality, risk, etc.)
- Weight each criterion (1-10)
- Score each option against criteria
- Which option has the highest weighted score?

**When to Use**: Multiple viable approaches, trade-off decisions, resource allocation

---

#### 3. **Pre-Mortem**
**Purpose**: Imagine failure to prevent it

**Questions to Ask:**
- Imagine it's 6 months from now and this project failed. What went wrong?
- What could we do now to prevent that failure?
- What are the early warning signs?
- What mitigation strategies should we build in?

**When to Use**: High-stakes projects, novel approaches, risk-heavy initiatives

---

#### 4. **Analogous Reasoning**
**Purpose**: Learn from similar situations

**Questions to Ask:**
- Has anyone done something similar before?
- What can we learn from that approach?
- What worked well? What didn't?
- How can we adapt those lessons to our context?

**When to Use**: When facing unfamiliar challenges, learning from best practices

---

#### 5. **Value Chain Analysis**
**Purpose**: Understand the flow of value creation

**Questions to Ask:**
- What's the sequence of activities needed?
- What adds the most value?
- Where can we optimize or eliminate steps?
- What dependencies exist between steps?

**When to Use**: Process design, workflow optimization, service delivery

---

### For Tasks.md

**Use when:**
- Breaking down work into actionable steps
- Sequencing activities
- Estimating effort and timeline

**Recommended Mental Models:**

#### 1. **Kanban Flow**
**Purpose**: Visualize work stages and flow

**Questions to Ask:**
- What are the major phases? (e.g., To Do, In Progress, Review, Done)
- What tasks belong in each phase?
- What's the logical sequence?
- Where might bottlenecks occur?
- What can be worked on in parallel?

**When to Use**: Task sequencing, workflow design, continuous delivery

---

#### 2. **OKR Framework** (Objectives and Key Results)
**Purpose**: Connect tasks to measurable outcomes

**Questions to Ask:**
- What's the objective (qualitative goal)?
- What are the key results (measurable outcomes)?
- What tasks move us toward each key result?
- How will we measure progress?

**When to Use**: Goal-oriented projects, measurable outcomes, team alignment

---

#### 3. **Pareto Principle** (80/20 Rule)
**Purpose**: Focus on highest-impact activities

**Questions to Ask:**
- Which 20% of tasks will deliver 80% of the value?
- What's the minimum viable version?
- What can we defer or eliminate?
- Where should we focus effort first?

**When to Use**: Resource constraints, MVP planning, prioritization

---

#### 4. **RACI Matrix**
**Purpose**: Clarify roles and responsibilities

**Questions to Ask:**
- For each task, who is:
  - **Responsible** (does the work)
  - **Accountable** (owns the outcome)
  - **Consulted** (provides input)
  - **Informed** (needs updates)
- Are responsibilities clear?
- Is any task missing an owner?

**When to Use**: Team projects, delegation, accountability clarity

---

#### 5. **Lean Canvas** (Simplified Business Model)
**Purpose**: Validate business model assumptions through tasks

**Questions to Ask:**
- Problem: What problem are we solving?
- Solution: What's our approach?
- Key Metrics: How do we measure success?
- Unique Value Prop: What makes this different?
- Channels: How do we reach users/customers?
- Revenue/Costs: What's the financial model?
- Key Tasks: What activities support each element?

**When to Use**: Business projects, startups, product development

## Elicitation Question Templates

### Opening Questions (All Phases)
- "Tell me more about [topic]..."
- "What's most important to you about [aspect]?"
- "What concerns you most about [area]?"
- "If you could wave a magic wand, what would [outcome] look like?"

### Clarifying Questions
- "Can you give me an example of that?"
- "What would success look like specifically?"
- "How would you know if this is working?"
- "What does [term] mean in your context?"

### Deepening Questions
- "Why is that important?"
- "What would happen if we didn't do that?"
- "What else should I know about [topic]?"
- "What am I not asking that I should be?"

### Constraint Questions
- "What's non-negotiable?"
- "What trade-offs are you willing to make?"
- "What's the timeline/budget/resource constraint?"
- "What dependencies exist?"

## Collaborative Writing Process

### Step 1: Elicit (Don't Prescribe)
- Ask questions based on selected mental models
- Listen to user's responses
- Probe deeper on unclear areas
- Summarize understanding

### Step 2: Draft Together
- "Based on what you've told me, here's a draft of [section]..."
- Present draft clearly
- Explicitly ask for feedback: "Does this capture it?"
- Note: You're writing WITH the user, not FOR the user

### Step 3: Iterate
- Make changes based on feedback
- Ask clarifying questions on changes
- Present updated version
- Repeat until user confirms: "This looks good!"

### Step 4: Confirm Explicitly
- "Are you happy with this [requirements/design/tasks]?"
- "Anything else to add or change?"
- "Should we move to the next section?"

**CRITICAL**: Do NOT proceed to next document until user explicitly confirms!

## Example Elicitation Flow

### For Requirements.md (Using First Principles + Stakeholder Mapping)

**AI**: "Let's start with requirements. I'll use first principles thinking to help us get to the core of what you need. First, can you tell me: what's the absolute core problem you're trying to solve with this project?"

**User**: "I need to organize my client proposals better."

**AI**: "Got it. If we stripped away all assumptions about how proposals are currently done, what are the fundamental requirements? What MUST happen?"

**User**: "I need to track which clients I've sent proposals to, what stage they're in, and follow up appropriately."

**AI**: "Perfect. Now let's think about stakeholders. Who will use this system besides you?"

**User**: "Just me, but my business partner might need to see status updates."

**AI**: "Understood. What does your business partner need to see specifically?"

**User**: "Just high-level: which proposals are pending, which are won, which are lost."

**AI**: "Great. Based on this, here's a draft of the requirements section:

[Presents draft requirements]

Does this capture everything? Anything to add or change?"

**User**: "Looks good, but I also need to store the actual proposal files."

**AI**: "Excellent point. Let me add that. Here's the updated version:

[Shows update]

How's this now?"

**User**: "Perfect!"

**AI**: "Wonderful! Since you confirmed this looks good, let's move to design.md where we'll figure out HOW to build this system..."
